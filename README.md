# relai

**조직을 이해하는 자동화.** 작업이 조직을 타고 스스로 굴러간다.

1~5인 스타트업을 위한 컴포저블 자동화 OS. 조직을 한 번 모델링하면, 반복되는 운영 글루(glue) 작업을 **액션 플랜**이 알아서 처리한다. 기존 자동화 툴(Zapier·n8n)이 평평한 `트리거 → 액션`인 것과 달리, relai는 **조직 구조 자체가 자동화가 올라타는 기반**이다. 누구에게/어느 팀으로 갈지가 조직 그래프로 자동 결정된다.

> rel**AI** — "relay"로 읽히고(신호가 노드에서 노드로 자동으로 넘어감), 동시에 글자에 AI가 박혀 있다.

---

## 1. 핵심 개념 (먼저 이것만 알면 됨)

- **노드(Node)** — 조직의 단위. 팀/역할/맥락. 노드 안에 노드가 들어갈 수 있다(`OriAlt > Product`, `OriAlt > Marketing`). 1~5인 팀에서는 인원이 아니라 **모자(맥락)**로 생각하면 된다.
- **멤버(Member)** — 노드에 속한 사람.
- **작업(Task)** — 할 일. 상태(`할 일 / 진행 중 / 완료`), 담당자, 소속 노드, 그리고 회의록 항목과의 연결을 가진다.
- **회의록(Checklist)** — 노드별 체크리스트. 작업이 끝나거나 자동화가 돌면 항목이 자동으로 열리거나 체크된다.
- **액션 플랜(Action Plan)** — 자동화 규칙. `트리거(언제) → 스텝(무엇을)` 구조. 각 플랜은 키가 필요한지(`BYOK`), MCP가 필요한지를 스스로 선언한다.
- **릴레이(Relay)** — 트리거가 발화해서 스텝이 조직을 타고 실행되는 그 순간. 발화될 때마다 토스트 + 릴레이 로그에 기록된다.

**트리거 종류**: 작업 상태 변경 · 협업 수락 · 수동 실행 · 스케줄 · 웹훅
**스텝 종류**: 회의록 체크 · 작업 생성/배정 · LLM 추출/요약/분류 · MCP 전달 · 알림

## 2. 화면 구성

3단 워크스페이스로 되어 있다.

- **왼쪽 — 구조/설정**
  - `조직` : 노드 안의 노드 + 멤버 칩.
  - `액션 플랜` : 5개 플랜 카탈로그. 트리거→스텝, BYOK/MCP 필요 여부, 켜고 끄는 토글.
- **가운데 — 실행(do)**
  - `협업 요청` (#2)
  - `노트 → 액션아이템` (#3)
  - `인바운드 → 태스크` (#5)
  - `작업` : 칸반 보드(할 일 / 진행 중 / 완료).
- **오른쪽 — 결과**
  - `스탠드업 다이제스트` (#4)
  - `회의록` : 자동 체크되는 항목들(새로 체크되면 청록색 펄스).
  - `릴레이 로그` : 모든 발화 기록.
  - `알림` : 인박스(미읽음 뱃지 + 모두 읽음).

오른쪽 아래에 뜨는 **토스트**는 릴레이가 발화될 때마다 잠깐 나타난다.

## 3. 시작하기

```bash
npm install
npm run dev        # http://localhost:5173/relai/
npm run dev:sync   # 위와 동일하되, Ctrl+C로 끌 때 자동 커밋 & 푸시
```

> base 경로가 `/relai/`라 `localhost:5173/` 가 아니라 **`localhost:5173/relai/`** 로 열린다.

`npm run dev:sync`로 돌리면 `dev.sh`가 vite를 띄우고, **Ctrl+C로 종료할 때 자동으로 `git add → commit → push`** 한다. 변경이 없으면 커밋을 건너뛴다.

## 4. LLM 키 설정 (BYOK)

#3·#4·#5는 LLM을 쓴다. **자기 API 키를 넣으면 브라우저에서 직접 호출**하기 때문에 서버리스/백엔드 없이 동작한다.

1. 가운데 `노트 → 액션아이템` 패널을 본다. 키가 없으면 키 입력 폼이 먼저 뜬다.
2. **제공자**(Anthropic / OpenAI), **모델**, **키**를 입력한다.
   - 기본 모델: Anthropic `claude-3-5-haiku-20241022`, OpenAI `gpt-4o-mini`.
   - 모델이 더 이상 안 먹으면 그 칸만 최신 모델 문자열로 바꾸면 된다.
3. **키 저장**을 누른다. 이후 패널 헤더의 **키 편집**으로 언제든 변경.

> 키는 이 브라우저(`localStorage`의 `relai.settings.v1`)에만 저장되고, **워크스페이스 내보내기(JSON)에는 절대 포함되지 않는다.** 호출 비용은 본인 키로 청구된다.

## 5. 5개 액션 플랜 사용법

키 없이 도는 것(#1, #2) → BYOK(#3, #4) → BYOK + MCP(#5) 순으로 난이도가 올라간다.

### #1 — 완료 → 체크 & 알림  ·  *키 불필요*
작업을 끝내면 회의록이 자동으로 정리되고 담당자에게 알림이 간다.

- **쓰는 법**: `작업` 보드의 카드에서 **완료 →** 버튼을 누른다. (할 일 카드는 **진행** 버튼도 있다.)
- **일어나는 일**: 그 작업에 연결된 회의록 항목이 자동 체크(청록 펄스) → 담당자와 소속 노드 멤버에게 알림 발화 → 릴레이 로그 기록 + 토스트.

### #2 — 협업 요청 라우터  ·  *키 불필요*
한 팀이 다른 팀에 일을 넘기면, 수락하는 순간 작업이 알아서 만들어진다.

- **쓰는 법**:
  1. `협업 요청` 패널에서 **보내는 노드 → 받는 노드**를 고른다.
  2. **요청 내용**(ex. "런칭 카피 작성")과 선택적으로 **추가 지시/프롬프트**를 적는다.
  3. **요청 보내기**. 대기 카드가 생긴다.
  4. 받는 쪽에서 **수락 →** (또는 거절).
- **일어나는 일(수락 시)**: 받는 노드에 작업이 생성되고 첫 멤버에게 배정 → 회의록 항목이 새로 열림 → 보내는/받는 양쪽 노드에 알림 발화.
- 데모용으로 **Product → Marketing "런칭 카피 작성"** 요청이 하나 미리 들어 있다. 바로 수락해보면 된다.
- 팁: 수락으로 생긴 작업을 다시 **완료** 처리하면 #1이 이어서 발화한다(두 플랜이 조직을 타고 연결되는 모습).

### #3 — 노트 → 액션아이템  ·  *BYOK*
회의록 텍스트를 붙여넣으면 LLM이 할 일을 뽑아 올바른 사람에게 꽂아준다.

- **쓰는 법**:
  1. (키 미설정이면 4번 항목대로 키 먼저 저장)
  2. 노드(어느 팀 회의록인지)를 고른다. **예시 채우기**로 샘플을 바로 넣을 수 있다.
  3. 회의록 텍스트를 붙여넣고 **액션아이템 추출**.
- **일어나는 일**: LLM이 **결정사항** + **액션아이템(담당자 추정)**을 뽑아낸다 → 각 액션아이템이 해당 노드의 작업으로 생성·배정되고 회의록 항목이 열림 → 담당자 알림. 추출 결과(결정사항/생성된 아이템)는 패널에 그대로 표시된다.

### #4 — 스탠드업 다이제스트  ·  *BYOK*
팀 현황을 한 장으로 요약해준다.

- **쓰는 법**: 오른쪽 위 `스탠드업 다이제스트` 패널에서 범위(**전체** 또는 특정 노드)를 고르고 **다이제스트 생성**.
- **일어나는 일**: 노드별 `완료 / 진행 중 / 대기` 작업과 최근 릴레이 발화를 모아 LLM이 `✅완료 / 🔄진행 / ⏭다음 + 한 줄 총평`으로 요약 → 다이제스트 카드로 쌓이고 전원 알림.
- 트리거는 "매일 스케줄"로 표시되지만, 정적 사이트엔 cron이 없어 **지금은 버튼(온디맨드)**이다. 진짜 자동 실행은 Phase 2.

### #5 — 인바운드 → 태스크  ·  *BYOK + MCP*
밖에서 들어온 메시지를 올바른 팀의 작업으로 분류하고 회신 초안까지 만든다.

- **쓰는 법**:
  1. `인바운드 → 태스크` 패널에서 **소스 유형**(문의/이메일/GitHub 이슈/기타)을 고른다. **예시 채우기** 가능.
  2. 들어온 메시지를 붙여넣고 **처리**.
- **일어나는 일**: LLM이 **담당 팀 · 우선순위 · 요약 · 회신 초안**으로 분류 → 올바른 노드에 작업 생성·배정·알림. 처리 카드에 우선순위 뱃지, **MCP 전달 예정 채널**, 회신 초안(+ 복사 버튼)이 뜬다.
- 한계: 실제 웹훅 수신 엔드포인트와 라이브 MCP 전달(자동 발송/Notion 생성)은 백엔드가 필요해 **지금은 시뮬레이션 + 초안 수동 복사**다. (Phase 2)

## 6. 보조 기능

- **액션 플랜 토글** : 각 플랜을 켜고 끌 수 있다. **자동 트리거 플랜(#1 완료, #2 협업 수락)은 끄면 발화하지 않는다.** #3·#4·#5는 패널 버튼으로 직접 실행하므로 토글과 무관하게 동작한다.
- **회의록** : 노드별로 자동 누적·체크되는 항목. 작업 완료·협업 수락·노트 추출·인바운드가 항목을 열거나 체크한다.
- **릴레이 로그** : 모든 발화를 `시간 · FIRE · 플랜명 · 컨텍스트 · 스텝`으로 기록.
- **알림** : 발화로 생긴 알림 인박스. 미읽음 개수 뱃지, 모두 읽음.

---

## 7. 데이터 & 워크스페이스

- 모든 워크스페이스 데이터는 브라우저 `localStorage`(`relai.state.v2`)에 저장된다. 백엔드 없음.
- 상단 툴바:
  - **내보내기** : 현재 워크스페이스를 `relai-workspace.json`으로 저장(**키는 미포함**).
  - **가져오기** : JSON을 불러와 워크스페이스 교체.
  - **초기화** : 시드 상태로 되돌림.
- 새 액션 플랜이 추가돼도 기존 워크스페이스를 날리지 않고 **플랜 카탈로그만 머지로 갱신**된다.

## 9. 코드 구조

```
src/
├── types.ts            데이터 모델 (Node, Member, Task, Checklist, ActionPlan,
│                       CollabRequest, Digest, Inbound, Trigger, Step …)
├── store.ts            상태 관리 훅(useRelaiStore) — seed, localStorage 영속,
│                       export/import, 모든 액션(완료/협업/노트/다이제스트/인바운드)
├── engine.ts           순수 실행기 — runPlansForTaskDone(#1),
│                       runPlansForCollabAccepted(#2), applyActionItems(#3),
│                       applyInbound(#5)
├── llm.ts              BYOK 브라우저 호출(공용 callLLM) — extractActionItems(#3),
│                       generateDigest(#4), triageInbound(#5)
├── settings.ts         LLM 키/제공자/모델 저장 (워크스페이스와 분리)
├── digest.ts           다이제스트용 작업 현황 스냅샷 빌더
├── App.tsx             레이아웃 + 토스트
└── components/
    ├── OrgPanel        조직 그래프
    ├── PlansPanel      액션 플랜 카탈로그 + 토글
    ├── CollabPanel     #2 협업 요청
    ├── NotesPanel      #3 노트 추출 + 키 설정 UI
    ├── DigestPanel     #4 다이제스트
    ├── InboundPanel    #5 인바운드 트리아지
    ├── TaskBoard       칸반 보드
    ├── ChecklistPanel  회의록(펄스)
    └── RelayLog        릴레이 로그 + 알림
```

엔진은 모두 **순수 함수**라(입력 상태를 받아 새 상태 조각을 반환) 테스트·확장이 쉽다. LLM이 필요한 #3·#4·#5만 비동기 호출 후 그 결과를 순수 함수에 넘긴다.

## 10. 솔직한 한계 (정적 사이트라서)

- **스케줄(#4)** : cron 실행 서버가 없어 앱이 켜져 있을 때 버튼으로만 실행.
- **웹훅 수신(#5)** : 외부 POST를 받을 엔드포인트가 없어 입력은 시뮬레이션.
- **MCP 라이브 전달(#5)** : 브라우저에서 사용자 MCP 서버를 직접 호출하기 어려워(인증·CORS) 지금은 채널 표시 + 초안 복사까지.

이 세 가지가 **Phase 2(얇은 백엔드)**의 핵심 과제다.
